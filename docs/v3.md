- [ ] `show_with_content` question flag — allows a question to embed the body of its linked content block so it can be shown alongside the question at display time (e.g. for data-interpretation or passage-based questions)
  - **Authoring format:** add `show_with_content: true` as an optional field in the question block header (after `tags:`, before question text); only valid when the question is gated on a content block (i.e. not ungated)
  - **Ingest:** when the flag is present, copy the nearest preceding `##` content block's body into a new `passage` field on the question at parse time; the question's unique ID / checksum must include the passage so that changes to the linked content block are detected and the question is re-ingested
  - **DB schema:** add `passage TEXT` (nullable) column to the `question` table
  - **Question model/controller:** include `passage` in all SELECTs and INSERTs
  - **App (CLI + web UI):** render `passage` as a blockquote or panel above the question text when the field is non-null
  - **Docs:** update `structured-question-format.md` and `subagent-format-reference.md` to document the flag and replace the current "embed passage in question text + ungated" workaround with this pattern
  - **`/review-course` skill:** Pass 2 agent prompt should check that any `show_with_content: true` question genuinely requires the passage to be readable (i.e. it is a data-interpretation or passage-dependent question, not just a recall question that happens to follow a content block), and that questions which do require a passage have the flag set

- [ ] Cloud hosting support with multi-user
  - Cloud is the only deployment mode that supports meaningful concurrency; local remains single-user / low-concurrency (current cron is fine there)
  - Deploy DB and API server to cloud (e.g. managed Postgres + containerised server)
  - Authentication: user accounts, session management
  - `user_id` derived from auth identity rather than local file
  - Cloud AI dispatch (`services/ai.js` TODO stub — support Lambda→Bedrock as alternative to local subprocess); Bedrock is required for cloud — local `claude` subprocess does not apply
  - Per-user data isolation enforced at API layer
  - Cron must be reworked for cloud: parallel user processing (concurrency-limited) and AI grading offloaded async to Lambda; extract cron to separate worker to allow horizontal scaling of the API server without duplicate ticks
  - `grade-ai` endpoint is synchronous (blocks until AI responds) — needs polling/webhook pattern for cloud; bites at ~10 concurrent users
  - `user_id` is an unenforced query param — data isolation must be enforced at API layer against auth identity; **security issue from user 2 onwards**
  - Rate limiting is global, not per-user — one heavy user can starve others; bites at ~2–3 concurrent active sessions (each session ~30–50 requests vs 100 req/15 min limit)
  - DB pool has no explicit `max` connections or query timeouts — default max is 10 connections; bites at ~10–20 concurrent requests
  - `POST /generate-adaptive` fires background AI tasks with no concurrency cap — needs a queue to prevent runaway Bedrock calls; bites at ~5–10 simultaneous session ends
  - Cron lag with freeText AI grading: ~5–10 active users; without AI (pure DB): ~20–50 users before ticks start skipping
  - Embedding model (~90 MB) loaded in-process — not suited to multiple server instances; only relevant at ~100+ concurrent users when horizontal scaling is needed
  - **Practical threshold: most concerns compound around 10–20 concurrent active users**
